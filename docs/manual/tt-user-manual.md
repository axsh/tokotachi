# tt ツール ユーザマニュアル

`tt`（旧称 `devctl`）は、異なるOS、エディタ、およびコンテナの組み合わせにまたがる、フィーチャー単位の開発環境を標準化・管理するためのマトリクス駆動型CLIツールです。

---

## 概要

`tt` は開発コンテナの起動、停止、接続、および後処理のワークフローを共通のインターフェースで提供します。
開発者は、ローカルの Git ワークツリー自動生成と DevContainer 等のコンテナ技術を組み合わせた一貫した環境で作業を行うことができます。

---

## 基本的な操作フロー

開発を開始し、コードを変更して、環境をクリーンアップするまでの基本的な流れは以下の通りです。

1. **環境の起動**: `tt up <feature>` コマンドで対象フィーチャーの開発コンテナを起動し、自動的に Git ワークツリーを作成します。
2. **コード編集**: `tt open <feature>` コマンドでエディタを起動し、コンテナ（DevContainer 等）に接続してコーディングを行います。
3. **環境の破棄**: 作業完了後、`tt down <feature>` または `tt close <feature>` コマンドでコンテナを破棄し、不要になったワークツリーをクリーンアップします。

---

## コマンドリファレンス

すべてのコマンドで `[branch]` 指定は任意です。省略した場合は、`<feature>` 名がそのままブランチ名として使用されます。

### 1. `up` — 開発コンテナの起動
対象フィーチャーの開発コンテナを起動し、対応する Git ワークツリーを自動生成します。

```bash
tt up <feature> [branch] [flags]
```

- **主なフラグ**:
  - `--editor <name>`: 起動するエディタ（または AI エージェント）を指定します。
    - 指定可能な値: `cursor`, `code` (VSCode), `ag` (Antigravity), `claude` (Claude Code)
  - `--ssh`: SSH モードを有効にします。
  - `--rebuild`: コンテナイメージを強制的に再ビルドします。
  - `--no-build`: コンテナイメージのビルドをスキップします。

### 2. `open` — エディタの起動
起動中の開発コンテナ、またはローカルのワークツリーに対して指定したエディタを開きます。

```bash
tt open <feature> [branch] [flags]
```

- **主なフラグ**:
  - `--editor <name>`: 使用するエディタを指定します。
  - `--attach`: 起動中のコンテナに対して DevContainer 経由でのアタッチを試みます（VSCode/Cursor のみ有効）。

### 3. `down` — コンテナの停止
対象フィーチャーの開発コンテナを停止し、削除します（Git ワークツリーは削除されません）。

```bash
tt down <feature> [branch]
```

### 4. `close` — 環境のクローズとクリーンアップ
開発コンテナを停止して削除し、対応する Git ワークツリーとローカルブランチも削除して環境を完全に破棄します。

```bash
tt close <feature> [branch] [flags]
```

- **主なフラグ**:
  - `--force`: ブランチがマージされていない場合でも、警告を無視して強制的に削除を行います。

### 5. `status` — ステータスの表示
対象フィーチャーの開発環境（コンテナの稼働状況やワークツリーの存在など）のステータスを表示します。

```bash
tt status <feature> [branch]
```

### 6. `shell` — コンテナ内シェルの起動
起動中の開発コンテナの内部で、インタラクティブシェル（bash 等）を起動します。

```bash
tt shell <feature> [branch]
```

### 7. `exec` — コンテナ内でのコマンド実行
起動中の開発コンテナの内部で、指定したコマンドを非インタラクティブに実行します。

```bash
tt exec <feature> [branch] -- <command> [args...]
```

### 8. `pr` — Pull Request の作成
GitHub CLI (`gh`) を使用して、対象フィーチャーの Pull Request を作成します。

```bash
tt pr <feature> [branch]
```

### 9. `list` — ブランチ一覧の表示
対象フィーチャーに関連するブランチや環境情報を一覧表示します。

```bash
tt list [flags]
```

- **主なフラグ**:
  - `--full`: 文字数制限によるカラムの省略（切り詰め）を行わず、すべての情報を表示します。
  - `--env`: 実行レポートに環境変数の設定セクションを含めます。

### 10. `scaffold` — テンプレートの適用
外部テンプレートリポジトリから定義ファイルをダウンロードし、プロジェクト構造やファイルを生成します。

```bash
tt scaffold [category] [name] [flags]
```

- **引数**:
  - `[category]`: テンプレートのカテゴリ（例: `project`, `feature`）。
  - `[name]`: テンプレートの名称（例: `axsh-go-standard`）。
  - ※カテゴリのみを指定して名前を省略した場合はエラーになります。
- **主なフラグ**:
  - `--yes`: `[y/N]` による実行前の最終確認プロンプトをスキップし、即座に適用します。
  - `--rollback`: 直前の scaffold 操作によって生成されたファイルを削除し、退避していた変更（stash）を復元します。
  - `--list`: 利用可能なテンプレート一覧を表示します。引数が1つ与えられた場合は、そのカテゴリでフィルタリングします。
  - `-v key=value`: テンプレートに渡すカスタムパラメータ（変数）の値を直接指定します。複数指定可能です。
  - `--default`: 必須ではない（任意）のパラメータ入力を促すプロンプトを表示せず、デフォルト値を自動適用します。
  - `--skip-deps`: 依存関係にある親テンプレートの自動解決と適用をスキップし、指定されたテンプレートのみを適用します。
  - `--force`: 過去のダウンロード履歴を無視し、すべてのテンプレートを強制的に再ダウンロードします。
  - `--repo <url>`: デフォルトのテンプレートリポジトリ URL を上書きします。
  - `--branch <name>`: テンプレート取得元の Git ブランチを上書きします。
  - `--lang <lang>`: テンプレートのローカライズ用ロケール（例: `ja`）を明示的に指定します。
  - `--root <path>`: 自動検出する Git ルートディレクトリの代わりに、指定したパスを適用先ルートとして使用します。

### 11. `prompt` --- プロンプトマニフェストの管理
プロンプトマニフェストのコンパイル、デプロイ、更新を行うサブコマンドグループです。

#### `prompt compile` --- マニフェストのコンパイル
プロンプトマニフェストとメモリドキュメントをコンパイルします。

```bash
tt prompt compile [flags]
```

- **主なフラグ**:
  - `--project <path>`: `project.yaml` のパスを指定します（デフォルト: `prompts/manifest/project.yaml`）。
  - `--target <name>`: コンパイル対象のターゲットを指定します（デフォルト: `TT_TARGET` 環境変数 または `all`）。
  - `--dry-run`: ファイルを書き出さず、結果を stdout に出力します。
  - `--apply`: 生成されたファイルをターゲットディレクトリに配置します。

#### `prompt deploy` --- コンパイルとデプロイ
コンパイルしたプロンプトファイルをターゲットディレクトリにデプロイします。ダイジェストベースの変更検知により、変更がない場合はスキップされます。

```bash
tt prompt deploy [flags]
```

- **主なフラグ**:
  - `--project <path>`: `project.yaml` のパスを指定します。
  - `--target <name>`: デプロイ対象のターゲットを指定します。
  - `--force`: 変更がない場合でも強制的にデプロイを実行します。
  - `--dry-run`: ファイルを書き出さず、実行内容を表示します。

#### `prompt update` --- 変更検知と自動更新
前回の更新以降にソースファイルに変更があったかを git log で検査し、変更があった場合のみコンパイルとデプロイを実行します。

```bash
tt prompt update [flags]
```

- **主なフラグ**:
  - `--project <path>`: `project.yaml` のパスを指定します。
  - `--target <name>`: 更新対象のターゲットを指定します。
  - `--force`: 変更検知をバイパスし、強制的に更新を実行します。
  - `--dry-run`: ファイルを書き出さず、実行内容を表示します。

#### ターゲット名称解決
`--target` オプションでは、以下の名前解決が行われます:

- **正式名**: `antigravity`, `cursor`, `claude-code`, `codex`
- **エイリアス**: `ag` / `agy` -> `antigravity`, `claude` -> `claude-code`
- **前方部分一致**: `anti` -> `antigravity`, `cur` -> `cursor`
- **特殊値**: `all` -- 全ターゲットに対して実行
- 候補が2つ以上にマッチする場合はエラーになります（例: `c` は `cursor` と `claude-code` に一致）

---

## グローバルフラグ

すべてのサブコマンドで利用可能な共通のフラグです。

- `--verbose`: デバッグレベルの詳細なログを出力します。
- `--dry-run`: 実際のファイル操作やコンテナ操作を実行せず、実行が予定されている処理内容のみをログに出力します。
- `--report <file>`: コマンド実行のサマリー（日付、フィーチャー、実行ステップ、環境変数、結果）を指定したマークダウンファイルに書き出します。

---

## 環境変数

`tt` ツールは、動作の調整や外部コマンドの呼び出しパスをオーバーライドするための環境変数を提供します。

### エディタとターゲットの解決

| 環境変数 | 説明 | デフォルト値 |
|---|---|---|
| `TT_EDITOR` | 使用するデフォルトエディタの上書き（`code`, `cursor` 等） | (自動検出) |
| `TT_TARGET` | `tt prompt` コマンドのデフォルトターゲット | `all` |

### 外部コマンドパスのオーバーライド

インストールパスが標準と異なる場合や、コマンド名を変更したい場合に設定します。

| 環境変数 | 説明 | デフォルト値 |
|---|---|---|
| `TT_CMD_CODE` | VSCode CLI のパス | `code` |
| `TT_CMD_CURSOR` | Cursor CLI のパス | `cursor` |
| `TT_CMD_AG` | Antigravity CLI のパス | `antigravity` |
| `TT_CMD_CLAUDE` | Claude Code CLI のパス | `claude` |
| `TT_CMD_GIT` | Git CLI のパス | `git` |
| `TT_CMD_GH` | GitHub CLI のパス | `gh` |
