# ttツールへのプロンプトコンパイル・デプロイ機能の追加

## 背景

本リポジトリ(tokotachi)には、サブリポジトリ vv5 由来のプロンプト管理基盤（`prompts/manifest/` および `prompts/memory/`）が既に存在している。vv5 では `agentctl` というコンパイラツールが以下の機能を提供している:

1. **compile**: manifest (YAML/Markdown) と memory ドキュメントを解析・バリデーション・解決し、`index.md` や `resolved manifest` を生成する
2. **deploy**: compile の成果物をターゲット別（antigravity, cursor, claude-code, codex）にエミットし、`.agent/`, `.cursor/` 等のルートフォルダに配備する
3. **validate**: manifest の構文・スキーマ検証
4. **check**: 配備済みファイルとコンパイル結果の差分(ドリフト)検知
5. **init**: ワークスペースの初期化（ttの`scaffold`に相当するため不要）

現在、本リポジトリでは `scripts/prompt/compile.sh` と `scripts/prompt/deploy.sh` が `agentctl` のラッパーとして存在するが、これを `tt` コマンドのサブコマンドとして組み込むことで、エージェント設定ファイル（`.agent/rules/`, `.agent/skills/`, `.agent/workflows/` 等）の自動生成・配備を `tt` から直接実行可能にする。

### 現状の課題

- `agentctl` は独立したバイナリで、PATHまたは`bin/`に配置が必要。ttとは別のモジュールである
- `_resolve_tool.sh` でagentctlを探索するが、見つからない場合はスキップされてしまう
- ttユーザにとって、agentctlのインストール・管理が追加負担になる
- manifest/memoryの変更後、AIエージェントが呼び出すスクリプトからttを利用できない
- `--editor` で使用するエディタ/エージェント名と `--target` で使用するターゲット名の名称体系が統一されておらず、「antigravity」は「ag」「agy」等コミュニティでも揺れがある
- 複数人の開発者が異なるCoding Agentを使っている場合、各自のタイミングでプロンプト設定を最新化する手段がない

## 要件

### 必須要件

#### R1. ターゲット名称解決の共通化

`--editor` と `--target` で使用する名称解決ロジックを共通パッケージとして実装する。

- **前方部分一致**: 入力文字列がターゲット名の前方に一致すれば解決する
  - 例: `anti` → `antigravity`, `cur` → `cursor`, `cl` → `claude-code`
- **曖昧性エラー**: 前方一致で候補が2つ以上ある場合はエラーとし、候補一覧を表示する
  - 例: `a` → エラー（`antigravity` と `all` が候補）
  - 例: `co` → エラー（`codex` と `cursor` が... ではなく `codex` のみ。`cursor` は `cu` 以降）。正確には `co` → `codex` のみ一致（`cursor` は `cu` で始まるため候補にならない）
- **特殊値 `all`**: 全ターゲットを意味する
  - `--target` のデフォルト値は `all`
  - `--editor` で `all` を指定した場合はエラーにする（エディタは1つに確定する必要がある）
- **エイリアス対応**: 既知のエイリアス（例: `ag` → `antigravity`, `vscode` → `code`）もサポート
- **正規名一覧**: `antigravity`, `cursor`, `claude-code`, `codex`, `all`
- **環境変数の分離**:
  - `--editor` のデフォルト: 環境変数 `TT_EDITOR`（既存）
  - `--target` のデフォルト: 環境変数 `TT_TARGET`（新規）。未設定時は `all`

既存の `pkg/detect/editor.go` にある `ParseEditor` / `ResolveEditor` と統合し、共通の名称解決関数を提供する。`--editor` は `all` 不可のバリデーションを追加する。

#### R2. `tt prompt compile` サブコマンドの実装

- `agentctl compile` と同等の機能を提供する
- `prompts/manifest/project.yaml` を起点にmanifestとmemoryドキュメントを解析・検証する
- 解析結果からターゲット別のファイルを生成する
- `--project` オプション: project.yaml のパス指定（デフォルト: `prompts/manifest/project.yaml`）
- `--dry-run` オプション: ファイル出力せず標準出力に結果を表示
- `--target` オプション: エミットターゲットの指定（デフォルト: `all`、環境変数 `TT_TARGET`）。前方部分一致で解決する
- `--apply` オプション: 生成ファイルをターゲットディレクトリに直接配置

#### R3. `tt prompt deploy` サブコマンドの実装

- `agentctl deploy` と同等の機能を提供する
- ソースのダイジェスト（SHA-256）を計算し、前回のダイジェストと比較して変更検知を行う
- 変更がなければスキップ（`--force` で強制再コンパイル）
- compile を内部で呼び出し、成功後にターゲットディレクトリにファイルを配備する
- `--project` オプション: project.yaml のパス指定
- `--target` オプション: エミットターゲットの指定（デフォルト: `all`、環境変数 `TT_TARGET`）。前方部分一致で解決する
- `--force` オプション: ダイジェストに関わらず強制再コンパイル
- `--dry-run` オプション: シミュレーション実行
- `--mode` オプション: ファイル配備モード（overwrite, immune, skip）

#### R4. `tt prompt update` サブコマンドの実装

`compile` + `deploy` をワンステップで実行する利便性コマンド。複数人が異なるCoding Agentを使う開発環境を想定し、各開発者が自分のタイミングでプロンプト設定を最新化できるようにする。

- **事前チェック**: ソースファイル（`prompts/manifest/` および `prompts/memory/`）に対して、前回の更新以降にgitで変更があったかを検知する。変更がなければ実行をスキップする
- `--force` オプション: 事前チェックを無視して強制的にcompile + deployを実行する
- `--target` オプション: 更新対象ターゲットの指定（デフォルト: `all`）
  - `all` の場合は全ターゲットを順に更新する
  - 個別ターゲット指定も可能
- `--project` オプション: project.yaml のパス指定
- `--dry-run` オプション: シミュレーション実行

##### 事前チェックの仕組み

各エミッターは、デプロイ成功時にターゲット固有のメタデータディレクトリに最終更新情報を書き出す。

**メタデータ配置先（ターゲット別）**:

| ターゲット | メタデータパス | 理由 |
|:---|:---|:---|
| antigravity | `.agent/.meta/last_update.yaml` | `.agent/` 配下で `rules/`, `skills/`, `workflows/` と同列だが隠しディレクトリなので影響なし。Antigravity（Gemini）は `.agent/` 直下のファイルを読み込むが、`.meta/` はマークダウンファイルでないため無視される |
| cursor | `.cursor/.meta/last_update.yaml` | `.cursor/` 配下。Cursorは `rules/` と `skills/` 以外を無視する |
| claude-code | `.claude/.meta/last_update.yaml` | `.claude/` 配下。Claude Codeは `rules/` 以外のサブディレクトリを無視する |
| codex | `.agents/.meta/last_update.yaml` | `.agents/` 配下。Codexは `AGENTS.md` とルール/スキルディレクトリのみを参照する |

**`last_update.yaml` のフォーマット**:
```yaml
# Auto-generated by tt prompt update. Do not edit.
updated_at: "2026-06-07T01:00:00+09:00"
source_digest: "sha256:abc123..."
target: antigravity
```

**変更検知ロジック**:
1. ターゲットのメタデータファイルを読む。存在しなければ「未更新」として更新対象とする
2. `updated_at` のタイムスタンプを取得する
3. `git log --since="<updated_at>" --name-only --pretty=format: -- prompts/manifest/ prompts/memory/` を実行し、タイムスタンプ以降に変更されたファイルがあるか確認する
4. 変更ファイルが存在すれば更新対象、存在しなければスキップ
5. `--force` の場合はこのチェックを全てバイパスする

#### R5. `scripts/prompt/compile.sh` と `scripts/prompt/deploy.sh` の書き換え

- 既存の `agentctl` 呼び出しを `tt prompt compile` / `tt prompt deploy` に置き換える
- `_resolve_tool.sh` を `tt` ツールを探索する形に変更する
- `scripts/prompt/update.sh` を新規追加（`tt prompt update` のラッパー）

#### R6. agentctl の内部パッケージの移植

- agentctl の `internal/compiler`, `internal/manifest`, `internal/memory`, `internal/emitter` パッケージのコードを ttモジュール内に移植する
- 独立した Go モジュール（`github.com/axsh/tokotachi/features/agentctl`）から、ttの依存範囲内で利用可能な形にする

#### R7. テンプレート変数の拡張（`{{target:*}}`）

既存のテンプレート変数システム（`{{kind:id}}`）に、コンパイル時の `--target` オプションを参照できるテンプレート変数を追加する。

- `{{target:name}}`: 現在のコンパイルターゲット名に展開される（例: `antigravity`, `cursor`）
- `{{target:meta_dir}}`: 現在のターゲットのメタデータディレクトリパスに展開される（例: `.agent/.meta/`）

これにより、プロンプト記述内でターゲット固有の文言を動的に切り替えることが可能になる。例えば、`architecture-maintainer` スキルが `./scripts/prompt/compile.sh` と `./scripts/prompt/deploy.sh --force` の2ステップを明記している箇所を、`./scripts/prompt/update.sh` に簡潔化する際にも活用できる。

#### R8. architecture-maintainer スキルの簡潔化

`tt prompt update` の導入に伴い、`prompts/manifest/code_content/capabilities/architecture-maintainer/skill.md` のワークフロー記述を更新する。

現状の記述:
```
you MUST run `./scripts/prompt/compile.sh` to compile the memory documents ...
If compilation succeeds without any errors, run `./scripts/prompt/deploy.sh --force` to deploy ...
```

更新後:
```
you MUST run `./scripts/prompt/update.sh --force` to compile and deploy the memory documents ...
```

compile + deploy の2ステップを `update` の1ステップに統合し、プロンプト記述をシンプルにする。

### 除外要件

- `agentctl init` コマンドの移植は不要（ttの`scaffold`が上位互換機能を持つ）
- `agentctl validate` / `agentctl check` の移植はこのフェーズでは対象外とする（将来的に追加可能だが、compile/deploy/updateが優先）

## 実現方針

### アーキテクチャ

agentctl の内部パッケージを `features/tt` のモジュール内に取り込む。agentctl 自体は `github.com/axsh/tokotachi/features/agentctl` として独立モジュールだが、主要ロジックは Go パッケージとして移植可能である。

```
features/tt/
  cmd/
    prompt.go          # [NEW] tt prompt サブコマンドの定義（compile/deploy/update）
  internal/
    prompt/             # [NEW] agentctlのロジック移植先
      compiler/         # compiler.go, config.go, deploy.go, digest.go, update.go
      manifest/         # types.go, parser.go, resolver.go, validator.go
      memory/           # frontmatter.go, indexer.go
      emitter/          # emitter.go, antigravity.go, cursor.go, claude_code.go, codex.go
                        # emit_mode.go, template.go, marker.go, limits.go

pkg/
  resolve/
    target.go           # [NEW] ターゲット名称解決の共通ロジック
    target_test.go      # [NEW]
```

### ターゲット名称解決の共通化

`pkg/resolve/target.go` に以下の共通関数を実装する:

```go
// ResolveTarget resolves a target name using prefix matching.
// Returns an error if the input is ambiguous (multiple matches).
// allowAll controls whether "all" is a valid target name.
func ResolveTarget(input string, allowAll bool) (string, error)

// ResolveTargets resolves a target name and returns the list of
// concrete target names. When input is "all", returns all known targets.
func ResolveTargets(input string) ([]string, error)
```

正規名一覧:
- `antigravity` (エイリアス: `ag`, `agy`)
- `cursor`
- `claude-code` (エイリアス: `claude`)
- `codex`
- `all` (全ターゲット)

前方部分一致の例:

| 入力 | 解決結果 | 理由 |
|:---|:---|:---|
| `antigravity` | `antigravity` | 完全一致 |
| `anti` | `antigravity` | 前方一致（一意） |
| `ag` | `antigravity` | エイリアス完全一致 |
| `cur` | `cursor` | 前方一致（一意） |
| `cl` | `claude-code` | 前方一致（一意） |
| `claude` | `claude-code` | エイリアス完全一致 |
| `co` | `codex` | 前方一致（一意。`cursor` は `cu` で始まる） |
| `a` | エラー | `antigravity` と `all` が候補 |
| `c` | エラー | `cursor`, `claude-code`, `codex` が候補 |
| `all` | `all` | 完全一致 |
| `al` | `all` | 前方一致（一意） |

既存の `pkg/detect/editor.go` の `ParseEditor` は、この共通名称解決を利用する形にリファクタリングする。エディタ名とターゲット名は同じ名前空間を共有する（例: `cursor` はエディタでありターゲットでもある）。

### コンパイルパイプライン（agentctlからの移植）

agentctlのコンパイルパイプラインは以下のステップで構成される:

1. **設定読み込み**: `prompts/manifest/project.yaml` をパース → `ProjectConfig`
2. **プロジェクトルート算出**: project.yaml のパスから2階層上をルートとする
3. **エンティティパース**: sources セクションの glob パターンに従い、YAML/Markdown ファイルを `Entity` に変換
4. **メモリドキュメントパース**: `memory_docs` パターンにマッチする Markdown の frontmatter を `MemoryDoc` に変換（GENERATED FILE バナー付きファイルはスキップ）
5. **スキーマバリデーション**: `prompts/manifest/schemas/` 配下の JSON Schema で検証
6. **ID一意性検証**: Entity と MemoryDoc 全体でIDの重複をチェック
7. **参照整合性検証**: depends_on 等の参照先が実際に存在するか検証
8. **解決 (Resolve)**: エンティティを kind 別にグループ化し `ResolvedManifest` を構築
9. **index.md 生成**: MemoryDoc からルーティングテーブルを含む index.md を生成
10. **Resolved Manifest 出力**: YAML として解決済みマニフェストを書き出し
11. **エミッター呼び出し**: ターゲット指定時、対応するエミッターで各種設定ファイルを生成。`--target all` の場合は全エミッターを順に実行

### デプロイパイプライン

1. **設定読み込み** → プロジェクトルート算出
2. **ターゲット解決**: `--target` の値を `ResolveTargets` で具体的なターゲットリストに変換
3. **ダイジェスト計算**: ソースファイル全体の SHA-256 ハッシュを計算
4. **各ターゲットについて**:
   a. **前回ダイジェストとの比較**: `tmp/dist/.compile-digest-{target}` に保存された値と比較
   b. **変更なし → スキップ** (force フラグで上書き可能)
   c. **compile 実行** (ターゲット指定 + apply 有効)
   d. **ダイジェスト保存**: コンパイル後に再計算したダイジェストを保存

### updateパイプライン

1. **ターゲット解決**: `--target` の値を具体的なターゲットリストに変換
2. **各ターゲットについて**:
   a. **事前チェック**: メタデータファイルの `updated_at` 以降にgitで変更があるか確認
   b. **変更なし → スキップ** (`--force` でバイパス可能)
   c. **compile + deploy 実行**
   d. **メタデータ更新**: `last_update.yaml` を書き出し

### エミッターの動作

各ターゲットのエミッターは `ResolvedManifest` を受け取り、ターゲット固有のファイル群を生成する:

| ターゲット | 出力先 | メタデータ | 内容 |
|:---|:---|:---|:---|
| antigravity | `.agent/rules/`, `.agent/skills/`, `.agent/workflows/` | `.agent/.meta/` | Policy -> rules, Capability -> skills, Procedure -> workflows |
| cursor | `.cursor/rules/`, `.cursor/skills/` | `.cursor/.meta/` | Policy -> rules (.mdc形式), Capability -> skills |
| claude-code | `.claude/rules/`, `.claude/skills/` | `.claude/.meta/` | Policy -> rules, Capability -> skills |
| codex | `.agents/rules/`, `.agents/skills/` | `.agents/.meta/` | Policy -> rules, Capability -> skills |

エミットモード:
- **overwrite**: 常に上書き（デフォルト）
- **immune**: 上書き + 対象ディレクトリ内の管理外ファイル（orphan）を削除
- **skip**: ファイルが存在しない場合のみ書き込み

### cobra コマンド構造

```
tt prompt
  tt prompt compile [--project PATH] [--target TARGET] [--dry-run] [--apply]
  tt prompt deploy  [--project PATH] [--target TARGET] [--force] [--dry-run] [--mode MODE]
  tt prompt update  [--project PATH] [--target TARGET] [--force] [--dry-run]
```

`tt prompt` をサブコマンドグループとし、`compile`, `deploy`, `update` を子コマンドとして配置する。

### スクリプト書き換え

`scripts/prompt/_resolve_tool.sh` を以下のように変更する:
- `agentctl` ではなく `tt` コマンドを探索する
- `TT_TOOL` 環境変数が設定されていればそれを使用
- `command -v tt` で PATH 上の tt を探索
- プロジェクトローカルの `bin/tt` をフォールバック

`scripts/prompt/compile.sh`:
```bash
exec "$TOOL" prompt compile "$@"
```

`scripts/prompt/deploy.sh`:
```bash
exec "$TOOL" prompt deploy "$@"
```

`scripts/prompt/update.sh` (新規追加):
```bash
exec "$TOOL" prompt update "$@"
```

### テンプレート変数の拡張

既存テンプレート変数 `{{kind:id}}` に加え、以下を追加する:

| 変数 | 展開例 | 用途 |
|:---|:---|:---|
| `{{target:name}}` | `antigravity` | 現在のコンパイルターゲット名 |
| `{{target:meta_dir}}` | `.agent/.meta/` | ターゲットのメタデータディレクトリ |

`--target all` でコンパイルする場合、テンプレート変数は各ターゲットのエミッター実行時にそれぞれのターゲット名に解決される。

### 依存関係

agentctl が使用している依存パッケージ:
- `gopkg.in/yaml.v3`: ttで既に使用中
- `github.com/santhosh-tekuri/jsonschema/v6`: スキーマバリデーション用（新規追加）
- `github.com/yuin/goldmark` + `github.com/yuin/goldmark-meta`: frontmatter パース用（新規追加）

ttの `go.mod` にこれらを追加する必要がある。

## 検証シナリオ

### シナリオ1: compile の基本動作

1. `tt prompt compile --project prompts/manifest/project.yaml --dry-run` を実行する
2. 標準出力に `index.md` の内容と resolved manifest が表示されることを確認する
3. ファイルが書き込まれないことを確認する

### シナリオ2: compile + apply（ターゲット指定）

1. `tt prompt compile --target anti --apply` を実行する（前方部分一致で `antigravity` に解決）
2. `.agent/rules/`, `.agent/skills/`, `.agent/workflows/` 以下にファイルが生成されることを確認する
3. 生成ファイルの内容が `prompts/manifest/` のソースファイルと対応していることを確認する

### シナリオ3: compile --target all

1. `tt prompt compile --target all --apply` を実行する
2. `.agent/`, `.cursor/`, `.claude/`, `.agents/` の全ターゲットディレクトリにファイルが生成されることを確認する

### シナリオ4: deploy の基本動作

1. `tt prompt deploy --target antigravity` を実行する
2. `.agent/` 以下にファイルが配備されることを確認する
3. 再度同じコマンドを実行し、「No changes detected. Skipping deploy.」と表示されることを確認する
4. `--force` を付けて実行し、再コンパイル・配備されることを確認する

### シナリオ5: deploy --dry-run

1. `tt prompt deploy --dry-run` を実行する
2. ファイルが実際に書き込まれないことを確認する
3. 「Deploy dry-run completed.」と表示されることを確認する

### シナリオ6: update の基本動作

1. `tt prompt update --target anti` を実行する
2. 初回は `last_update.yaml` が存在しないため、compile + deploy が実行されることを確認する
3. `.agent/.meta/last_update.yaml` が作成されることを確認する
4. 再度 `tt prompt update --target anti` を実行する
5. ソースに変更がないため「No changes detected. Skipping update.」と表示されることを確認する
6. `prompts/memory/` 以下のファイルを変更し、`git add && git commit` する
7. 再度 `tt prompt update --target anti` を実行する
8. 変更が検知されて compile + deploy が実行されることを確認する

### シナリオ7: update --target all

1. `tt prompt update --target all` を実行する
2. 全ターゲット（antigravity, cursor, claude-code, codex）に対して順にupdate処理が実行されることを確認する
3. 各ターゲットの `.meta/last_update.yaml` が更新されることを確認する

### シナリオ8: update --force

1. 既に最新状態であることを確認する（事前チェックでスキップされる状態）
2. `tt prompt update --force --target anti` を実行する
3. 事前チェックをバイパスして compile + deploy が実行されることを確認する

### シナリオ9: ターゲット名の前方部分一致

1. `tt prompt compile --target anti --dry-run` → 正常に `antigravity` に解決される
2. `tt prompt compile --target cur --dry-run` → 正常に `cursor` に解決される
3. `tt prompt compile --target a --dry-run` → エラー: 「ambiguous target "a": matches [all, antigravity]」
4. `tt prompt compile --target c --dry-run` → エラー: 「ambiguous target "c": matches [claude-code, codex, cursor]」
5. `tt prompt compile --target ag --dry-run` → 正常にエイリアスで `antigravity` に解決される

### シナリオ10: スクリプト経由の実行

1. `scripts/prompt/compile.sh --dry-run` を実行する
2. 内部で `tt prompt compile --dry-run` が呼び出されることを確認する
3. `scripts/prompt/update.sh --target anti` を実行する
4. 内部で `tt prompt update --target anti` が呼び出されることを確認する

### シナリオ11: バリデーションエラー時の動作

1. 不正な manifest ファイル（IDなし等）を一時的に作成する
2. `tt prompt compile` を実行する
3. バリデーションエラーが表示され、ファイル生成が行われないことを確認する
4. 終了コードが非ゼロであることを確認する

### シナリオ12: 環境変数によるデフォルト値

1. `TT_TARGET=cursor tt prompt deploy` を実行する
2. cursor ターゲットのみにデプロイされることを確認する
3. 環境変数未設定で `tt prompt deploy` を実行する
4. デフォルト値 `all` により全ターゲットにデプロイされることを確認する
