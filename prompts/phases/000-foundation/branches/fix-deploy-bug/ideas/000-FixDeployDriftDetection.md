# 要求仕様書: デプロイ時のドリフト（差分）検出に基づくデプロイ判定の改善

## 1. 背景 (Background)

現在、`tt prompt deploy` および `tt prompt update` のデプロイ処理におけるスキップ判定は、入力ソースコードのハッシュ値のみをキャッシュキー（`tmp/dist/.compile-digest-<target>` に記録された値）として比較しています。

この仕様には以下の問題（バグ）が存在します。

1. **デプロイ先ファイルが変更・削除された場合（成果物の欠落）**:
   Git チェックアウト直後などでデプロイ先（`.agents/` など）のファイル群が空、もしくは手動で削除・変更された状態であっても、入力ソースコードが変更されていなければ「変更なし」と判定され、デプロイがスキップされてしまいます。
2. **手動コンパイルとの不整合**:
   `tt prompt compile` を手動で実行して `tmp/dist/` 配下のファイルが書き換わった場合でも、入力ソースコードが変わっていなければ、`tt prompt deploy` はデプロイ処理を行わず、最新の成果物がデプロイ先に反映されません。

したがって、本来「入力ソースコードが変更された」場合だけでなく、「デプロイ先の実ファイル状態が期待される状態（Resolved Manifest に基づく期待値）と異なる」場合も、デプロイ（コンパイルおよび配置）をスキップせずに確実に実行する必要があります。

## 2. 要件 (Requirements)

1. **デプロイ先検証によるスキップ判定の強化**:
   `tt prompt deploy` 実行時、入力ソースコードのハッシュ値が一致している場合であっても、デプロイ先の実ファイル群と期待されるファイル状態（Resolved Manifest）の比較検証（ドリフト検出）を実行すること。
2. **ドリフト（不整合）検出時の強制デプロイ**:
   デプロイ先の検証結果、ファイルの欠落、手動変更、古いバージョンのファイルの存在（ドリフト）を検出した場合は、`--force` オプションが指定されていなくても自動的にコンパイルおよび配置（デプロイ）処理を実行すること。
3. **無変更かつ整合時のスキップ**:
   入力ソースコードのハッシュ値が一致し、かつデプロイ先の実ファイル群にドリフトが検出されない場合に限り、デプロイ処理をスキップすること。

## 3. 実現方針 (Implementation Approach)

### 3.1. デプロイ処理 (`deploy.go`) の修正

deploy.go の Deploy 関数において、以下の改善を行います。

1. スキップ判定の条件式を変更します。入力ハッシュの一致確認だけでは即座にスキップせず、デプロイ先のドリフト検出処理を追加します。
2. ハッシュ値が一致している場合、以下の手順でデプロイ先の検証（Check）を実行します。
   - プロジェクト設定で定義された Resolved Manifest ファイル（cfg.Outputs.ResolvedManifest）を読み込みます。
   - 読み込んだ YAML データを manifest.ResolvedManifest 構造体にデシリアライズ（Unmarshal）します。ファイルの読み込みやパースに失敗した場合は、スキップせずにデプロイを続行（コンパイルおよび配置を実行）します。
   - 指定されたターゲット（opts.Target。未指定時は antigravity）に対応するエミッターを初期化します。
     - antigravity : emitter.NewAntigravityEmitter(rootDir)
     - cursor : emitter.NewCursorEmitter(rootDir)
     - claude-code : emitter.NewClaudeCodeEmitter(rootDir)
     - codex : emitter.NewCodexEmitter(rootDir)
   - 初期化したエミッターの Check(resolved, buildDir) メソッドを呼び出します。
   - Check が true を返し、かつエラーが発生しなかった場合のみ、デプロイ処理をスキップします。それ以外（Check が false またはエラー発生時）は、デプロイを強制実行します。

## 4. 検証シナリオ (Verification Scenarios)

### シナリオ1: デプロイ先ファイルが手動で削除された場合（欠落の検出）

1. ソースコードに変更がない状態で、既に一度デプロイが完了している環境を準備します。
2. デプロイ先（例: .agents/skills/ 内のファイル）を1つ手動で削除します。
3. --force を付けずに tt prompt deploy を実行します。
4. 期待される結果: デプロイがスキップされず、コンパイルと配置処理が実行され、削除したファイルが再生成されること。

### シナリオ2: デプロイ先ファイルが手動で書き換えられた場合（変更の検出）

1. ソースコードに変更がない状態で、既に一度デプロイが完了している環境を準備します。
2. デプロイ先のファイルの1つを手動で編集して内容を書き換えます。
3. --force を付けずに tt prompt deploy を実行します。
4. 期待される結果: デプロイがスキップされず、コンパイルと配置処理が実行され、ファイルが期待される内容に復元されること。

### シナリオ3: ソースもデプロイ先も変更がない場合（スキップの動作）

1. デプロイ完了後、ソースファイルもデプロイ先ファイルも一切変更しない状態で、tt prompt deploy を実行します。
2. 期待される結果: デプロイ処理がスキップされること（Skipped: true）。

1. deploy_test.go への単体テストの追加:
   - TestDeploy_Drift テストケースを追加します。
   - 1回目の Deploy 実行後、エミッターの出力先のファイルを削除、または内容変更します。
   - 2回目の Deploy を --force=false で実行し、result.Skipped が false になること、およびファイルが再生成されることを検証します。

2. テスト実行:
   - go test -v ./features/tt/internal/prompt/compiler/... を実行し、すべてのテスト（既存テストおよび新規の TestDeploy_Drift）がパスすることを確認します。
