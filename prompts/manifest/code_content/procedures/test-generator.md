# テスト実装計画作成ワークフロー

アイデア/仕様書 (`.../ideas/.../XXX-{Name}.md`) を元に、**要件ごとの実現性を検証するためのテスト実装計画書** (`.../plans/.../YYY-{Name}-TestPlan.md`) を作成する。

`/create-implementation-plan` が機能実装の計画を立てるのに対し、このワークフローは「最低限の実装で要件が実現できたと言い切れるか」を検証するためのテストに特化した計画を立てる。

## 1. 入力とルールの確認

1.  **入力ファイルの特定**:
    *   ユーザーが指定したファイル、または現在エディタで開いているファイルを「仕様書」として扱う。
2.  **ルールの読み込み**:
    *   `{{policy:testing-rules}}` を読み込む。
3.  **ステータスの取得**:
    *   `scripts/utils/show_current_status.sh` を実行し、`phase`, `branch`, `next_plan_id` を取得する。

## 2. 出力先の決定

*   **出力ディレクトリ**: `prompts/phases/[Phase]/branches/[Branch]/plans/`
*   **ファイル名**: `[NextID]-[名前]-TestPlan.md` (`-TestPlan` で通常の実装計画書と区別)

## 3. 仕様書の分析

仕様書を精読し、背景・目的、アーキテクチャ、データフロー、外部依存関係、既存コードとの統合ポイントを把握する。

## 4. 要件の抽出

仕様書から**全ての要件**を漏れなく抽出し、番号付きリストとして整理する。

*   各要件に一意の識別子を付与する（例: `REQ-001`, `REQ-002`）。
*   分類: **機能要件** / **非機能要件** / **統合要件**
*   仕様書に明示されていない暗黙的な要件（エラーハンドリング、ログ出力等）も抽出する。

## 5. 実現根拠の列挙

各要件について、「この要件が実現できたと言い切れる根拠」を**複数**列挙する。

> [!IMPORTANT]
> 「テストが通ること」ではなく、「何がどう観測・計測できること」が根拠になるかを明確にすること。

## 6. 確認手段の列挙

各根拠について、「その根拠を得るために具体的にやるべきこと」を以下の視点から網羅的に列挙する:

| 視点 | 例 |
| :--- | :--- |
| ログ確認 | 特定のログレベル・メッセージが出力されること |
| API応答確認 | HTTPステータスコード・ボディの検証 |
| ファイル出力確認 | 生成ファイルの存在・内容の検証 |
| プロセス確認 | プロセスの起動・稼働・停止 |
| 通信確認 | コンポーネント間通信の実行確認 |
| データ確認 | 内部状態が期待通りに更新されること |
| エラー確認 | 異常系の適切なハンドリング |

## 7. 確認手順の具体化

各確認手段について、**別の開発者がそのまま実行できるレベル**の詳細な手順を記述する。

各手順に含めるべき項目: **前提条件** / **入力** / **操作手順** / **期待結果** / **判定基準**

## 8. テストシナリオへの変換

確認手順をテスト種別に分類し、実装計画としてまとめる。
テスト種別の配置先やファイル命名規則は `{{policy:testing-rules}}` Section 4 を参照。

テストケースの記述形式:

```markdown
#### TC-XXX: [テストケース名]

*   **対応要件**: REQ-XXX
*   **対応根拠**: [根拠の参照]
*   **テスト種別**: 単体テスト / 統合テスト / E2Eテスト
*   **配置先**: [ファイルパス]
*   **テスト関数名**: `TestXxx_Yyy`
*   **テストシナリオ**:
    1. [Arrange] テストデータの準備
    2. [Act] テスト対象の実行
    3. [Assert] 期待結果の検証
*   **実装メモ**: [モック対象、ヘルパー関数など]
```

## 10. セルフレビューと完了確認

1.  **要件網羅性**: 仕様書の全要件が抽出されているか。暗黙的な要件も含まれているか。
2.  **根拠の十分性**: 各要件に複数の根拠が列挙されているか。観測可能な事象が根拠になっているか。
3.  **確認手段の多角性**: Section 6 の視点一覧を参照し、該当する視点を漏れなくカバーしているか。
4.  **手順の具体性**: 各確認手順が「別の開発者がそのまま実行できるレベル」の詳細さを持っているか。
5.  **テスト分類の妥当性**: テスト種別の分類が適切か。
6.  **カバレッジの確認**: 要件カバレッジマトリクスに空白がある場合、その理由が妥当か。
7.  **`/execute-implementation-plan` との互換性**: チェックボックス付きの実装ガイドが含まれているか。

## 11. ファイルの保存と完了

1.  作成したファイルを指定されたディレクトリに保存する。
2.  作成したファイルへのリンクをユーザーに提示する（`file://` 形式）。
3.  **フェーズ移行禁止**: 次のステップに進むのは、ユーザーが明示的に指示した場合のみ。
