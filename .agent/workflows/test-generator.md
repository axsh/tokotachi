# テスト実装計画作成ワークフロー

このワークフローは、アイデア/仕様書 (`.../ideas/.../XXX-{Name}.md`) を元に、**要件ごとの実現性を検証するためのテスト実装計画書** (`.../plans/.../YYY-{Name}-TestPlan.md`) を作成します。

`/create-implementation-plan` が機能実装の計画を立てるのに対し、このワークフローは「最低限の実装で要件が実現できたと言い切れるか」を検証するためのテストに特化した計画を立てます。

## 1. 入力とルールの確認

1.  **入力ファイルの特定**:
    *   ユーザーが指定したファイル、または現在エディタで開いているファイルを「仕様書」として扱います。
    *   ファイル名の形式は通常 `[3桁の連番]-[名前].md` です。
2.  **ルールの読み込み**:
    *   `.agent/rules/testing-rules.md` を読み込みます。
3.  **ステータスの取得**:
    *   `scripts/utils/show_current_status.sh` を実行します。
    *   JSON出力から `phase`, `branch`, `next_plan_id` を取得します。
    *   以下、`[Phase]`, `[Branch]`, `[NextID]` とします。

## 2. 出力先の決定

1.  **出力ディレクトリの特定**:
    *   **Phase-Aware Path**: `prompts/phases/[Phase]/branches/[Branch]/plans/` を出力先とします。
    *   このディレクトリが存在しない場合は作成します。
2.  **ファイル名の決定**:
    *   形式: `[NextID]-[名前]-TestPlan.md`
    *   `-TestPlan` サフィックスにより、通常の実装計画書と区別します。

## 3. 仕様書の分析 (Step 1: 仕様書の読み込み)

仕様書を精読し、以下の情報を正確に把握します:

*   背景と目的
*   アーキテクチャおよびコンポーネント構成
*   データフロー
*   外部依存関係（API、プロセス間通信、ファイルシステムなど）
*   既存コードとの統合ポイント

## 4. 要件の抽出 (Step 2: 要件リストの作成)

仕様書から**全ての要件**を漏れなく抽出し、番号付きリストとして整理します。

*   各要件には一意の識別子を付与します（例: `REQ-001`, `REQ-002`, ...）。
*   要件は以下の粒度で分類します:
    *   **機能要件**: 「Xが入力されたらYが出力される」等の具体的な振る舞い
    *   **非機能要件**: パフォーマンス、耐障害性、セキュリティなど
    *   **統合要件**: 外部システムやコンポーネント間の連携に関するもの
*   仕様書に明示されていないが暗黙的に求められる要件（例: エラーハンドリング、ログ出力）も抽出してください。

## 5. 実現根拠の列挙 (Step 3: 要件ごとの検証基準)

各要件について、「この要件が実現できたと言い切れる根拠」を**複数**列挙します。

> [!IMPORTANT]
> 根拠は「何が確認できれば実現できたと判断するか」という観点で記述してください。
> 単に「テストが通ること」ではなく、「何がどう観測・計測できること」が根拠になるかを明確にしてください。

根拠の例:
*   特定のログメッセージが出力されること
*   特定のHTTPステータスコードが返却されること
*   特定のファイルが生成されること
*   特定のプロセスが起動・停止すること
*   特定のデータ構造が期待通りの値を持つこと
*   特定の通信が成功すること
*   特定のエラーが適切にハンドリングされること
*   特定のパフォーマンス基準を満たすこと

## 6. 確認手段の列挙 (Step 4: 根拠を得るためにやるべきこと)

Step 3 で列挙した根拠ごとに、「その根拠を得るために具体的にやるべきこと」を**複数の視点**から列挙します。

以下の視点を網羅的にチェックし、該当するものを全て列挙してください:

| 視点 | 確認手段の例 |
| :--- | :--- |
| **ログ確認** | 特定のログレベル・メッセージが出力されることを確認する |
| **API応答確認** | HTTPリクエストを送信し、レスポンスのステータスコード・ボディを検証する |
| **ファイル出力確認** | 生成されたファイルの存在・内容・フォーマットを検証する |
| **プロセス確認** | プロセスの起動・稼働・停止を確認する |
| **通信確認** | コンポーネント間やシステム間の通信が実際に行われることを確認する |
| **データ確認** | データベースや内部状態が期待通りに更新されることを確認する |
| **スクリプト実行** | 検証用の小さなスクリプトやプログラムを作成・実行して動作を確認する |
| **エラー確認** | 異常系の入力や障害を意図的に発生させ、適切なエラーハンドリングを確認する |
| **UI/出力確認** | スクリーンショットやコンソール出力を通じて表示内容を確認する |
| **パフォーマンス確認** | 処理時間・メモリ使用量・スループットなどを計測する |

## 7. 確認手順の具体化 (Step 5: 手順の明確化)

Step 4 で列挙した各確認手段について、**具体的な実行手順**を明確にします。

> [!IMPORTANT]
> **具体性の基準**: この手順を読んだ別の開発者（またはAIエージェント）が、追加の情報なしにそのまま実行できるレベルの詳細さが必要です。

各手順には以下を含めてください:

1.  **前提条件**: どのサービスやプロセスが起動している必要があるか
2.  **入力**: 何を入力として準備するか（テストデータ、設定ファイル、環境変数など）
3.  **操作手順**: 具体的に何をするか（コマンド、API呼び出し、操作手順）
4.  **期待結果**: 何がどこにどのように現れるか（ログの出力先・内容、レスポンスの形式など）
5.  **判定基準**: 成功/失敗をどう判定するか

手順の記述例:
```
### ログ出力の確認手順
1. 前提条件: サーバーが `go run ./cmd/server` で起動していること
2. 入力: テスト用のリクエストボディ (JSON形式)
3. 操作:
   a. curl -X POST http://localhost:8080/api/v1/chat -d '{"message": "test"}' -H "Content-Type: application/json"
4. 期待結果:
   - サーバーの標準出力に `[INFO] chat request received: message="test"` が出力される
   - レスポンスのHTTPステータスが 200 である
5. 判定基準: ログメッセージのフォーマットと内容が仕様書の定義と一致すること
```

## 8. テストシナリオへの変換 (Step 6: テスト分類と実装計画)

Step 7 で具体化した確認手順を、テストの種別に分類し、実装計画としてまとめます。

### 8.1 テスト種別への分類

各確認手順を以下のテスト種別に分類してください:

| テスト種別 | 配置先 | 判断基準 |
| :--- | :--- | :--- |
| **単体テスト** | 対象パッケージの `_test.go` | 外部依存なしに関数・メソッド単位で検証可能なもの |
| **統合テスト** | `tests/[category]/xxx_test.go` | 外部システムやプロセス間連携の検証が必要なもの |
| **E2Eテスト** | `tests/[category]/e2e_xxx_test.go` | エンドツーエンドのシナリオ検証が必要なもの |
| **検証スクリプト** | `scripts/verify/xxx.sh` | 一時的な動作確認やデバッグ用のスクリプト |

> [!NOTE]
> テスト種別の配置先はプロジェクトの既存構成に合わせてください。
> 上記は一般的な例であり、実際の配置先は既存のテストファイルの配置パターンを調査して決定してください。

### 8.2 テストシナリオの記述

各テストケースについて、以下の形式で記述してください:

```markdown
#### TC-XXX: [テストケース名]

*   **対応要件**: REQ-XXX
*   **対応根拠**: [Step 3 で列挙した根拠の参照]
*   **テスト種別**: 単体テスト / 統合テスト / E2Eテスト / 検証スクリプト
*   **配置先**: [ファイルパス]
*   **テスト関数名**: `TestXxx_Yyy` (Goの場合)
*   **前提条件**: [テスト実行前に必要な準備]
*   **テストシナリオ**:
    1. [Arrange] テストデータの準備: ...
    2. [Act] テスト対象の実行: ...
    3. [Assert] 期待結果の検証: ...
*   **実装メモ**: [モック対象、テストヘルパーの利用、テストフィクスチャなど]
```

### 8.3 テストの依存関係と実行順序

テストケース間の依存関係を整理し、実行順序を決定します:

1.  **単体テスト**: 最も基礎的な検証。他のテストの前提となる。
2.  **統合テスト**: 単体テストが通過した上で、コンポーネント間の連携を検証。
3.  **E2Eテスト**: 統合テストが通過した上で、エンドツーエンドのシナリオを検証。

## 9. テスト実行計画 (Execution Plan)

テスト実装計画の最後に、テストを実行するための具体的な手順を記載します。

### 9.1 実装手順

テストコードの実装順序を、ファイル単位で時系列に沿って記述します:

```markdown
## Step-by-Step Implementation Guide

1.  **[ステップ名]**:
    *   Create/Edit `[ファイルパス]`
    *   [具体的な実装内容]
    *   `git add && git commit -m "test: [コミットメッセージ]"`

2.  **[ステップ名]**:
    *   ...
```

### 9.2 テスト実行手順

```markdown
## Test Execution Plan

1.  **ビルドと単体テスト**:
    ```bash
    ./scripts/process/build.sh
    ```

2.  **統合テスト (選択的実行)**:
    ```bash
    ./scripts/process/integration_test.sh --categories "[category]" --specify "[対象テスト名]"
    ```

3.  **全テスト実行 (リグレッション確認)**:
    ```bash
    ./scripts/process/integration_test.sh --categories "[category]"
    ```
```

> [!IMPORTANT]
> この計画書の出力は `/execute-implementation-plan` ワークフローの入力として使用されます。
> `/execute-implementation-plan` で実装・実行・修正ループが回せるように、十分な具体性を持たせてください。

## 11. セルフレビューと完了確認

計画書を完了とみなす前に、以下の観点でセルフレビューを行い、必要に応じて修正してください。

1.  **要件網羅性チェック**: 仕様書の全要件が `要件一覧` に抽出されているか。暗黙的な要件も含まれているか。
2.  **根拠の十分性チェック**: 各要件について、複数の根拠が列挙されているか。「テストが通ること」だけでなく、観測可能な事象が根拠になっているか。
3.  **確認手段の多角性チェック**: Step 4 の視点一覧を参照し、該当する視点を漏れなくカバーしているか。
4.  **手順の具体性チェック**: 各確認手順が「別の開発者がそのまま実行できるレベル」の詳細さを持っているか。
    *   コマンド、ファイルパス、期待出力が具体的に記述されているか。
    *   「適切に確認する」「正しく動作することを確認する」といった曖昧な記述がないか。
5.  **テスト分類の妥当性チェック**: テスト種別（単体/統合/E2E）の分類が適切か。外部依存のないテストが統合テストに分類されていないか。
6.  **カバレッジの確認**: 要件カバレッジマトリクスに空白がある場合、その理由が妥当か。
7.  **実行順序の確認**: テスト間の依存関係が正しく反映され、実行順序が適切か。
8.  **`/execute-implementation-plan` との互換性チェック**: 作成した計画書が `/execute-implementation-plan` ワークフローの入力として利用可能な形式になっているか。特に `Step-by-Step Implementation Guide` のチェックボックスが含まれているか。

## 12. ファイルの保存と完了

1.  作成したファイルを指定されたディレクトリに保存します。
2.  作成したファイルへのリンクをユーザーに提示します。
    *   「file://プロジェクトルートからの相対パス」を用いること。
3.  **フェーズ移行**: 次のステップ（テスト実装の実行等）に進むのは、ユーザーが明示的に指示した場合のみです。勝手に `/execute-implementation-plan` に進まないでください。
